API errors: status codes, retries, and safe recovery

How to decide whether an API request should be corrected, delayed, retried, or escalated.

This guide favors observable evidence, reversible tests, and official product documentation. Commands are examples; adapt names and paths to the environment you are responsible for.

01

Classify the response before retrying

HTTP status families describe who can make progress. A 400 usually means the request must change. A 401 or 403 requires a credential, permission, project, or policy check. A 429 says the caller has exceeded a limit. A 500, 503, or provider-specific overload response can be transient, but only if the request is otherwise valid.

Always preserve the response body, status code, request ID, and retry-related headers after removing secrets. Provider SDKs often wrap this information, so inspect the structured error rather than relying on the exception title alone.

  • Do not retry malformed 400 requests unchanged.
  • Do not rotate credentials blindly for a project-level quota error.
  • Treat authentication and authorization as different problems.
  • Use provider status pages to distinguish local failure from a service incident.
02

Retry only operations that are safe to repeat

A read request is commonly safe to repeat. A request that creates a charge, sends a message, or mutates data may run twice if the first response is lost after the server completes the work. Before adding automatic retries, determine whether the endpoint is idempotent or supports an idempotency key.

Bound every retry loop by both attempt count and elapsed time. Exponential backoff with jitter spreads clients across time and reduces synchronized retry spikes. Respect Retry-After or provider-specific delay guidance when present, and stop retrying when the request deadline can no longer be met.

  • Cap attempts and maximum delay.
  • Add random jitter instead of fixed sleep intervals.
  • Keep the same idempotency key for retries of one logical operation.
  • Record the final failure rather than swallowing it.
03

Reduce load before requesting more quota

A quota increase does not repair an application that retries every failure immediately. Measure request rate, token or payload volume, concurrency, cache hit rate, and duplicate work. Queue bursts and apply backpressure so callers cannot create unlimited in-flight work.

When a provider exposes several limits, identify the exact one named in the error. Requests per minute, tokens per minute, daily requests, concurrent jobs, and account spending are separate constraints. A new API key in the same project normally does not reset a project-scoped limit.

  • Aggregate identical in-flight requests where possible.
  • Cache stable responses with an explicit freshness policy.
  • Set per-user and global concurrency limits.
  • Alert on sustained retry rate, not only final failures.
04

Escalate with evidence a provider can use

Persistent server errors need a narrow incident report. Include UTC timestamps, request IDs, region, model or endpoint, SDK version, status code, sanitized response body, and whether the same minimal request fails from another environment.

Never include API keys, authorization headers, full customer payloads, or private prompts in a public issue. Reproduce with synthetic data and provide hashes or sizes when the original content matters to the failure.
