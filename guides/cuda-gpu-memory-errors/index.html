GPU and CUDA errors: memory, drivers, and runtime compatibility

A practical way to distinguish capacity problems from driver, toolkit, package, and container mismatches.

This guide favors observable evidence, reversible tests, and official product documentation. Commands are examples; adapt names and paths to the environment you are responsible for.

01

Prove the GPU is visible at each layer

GPU software crosses several layers: hardware, host driver, container runtime when used, language package, and application. Test visibility at each boundary. A host nvidia-smi result does not prove a container can access the device, and a CUDA toolkit installed on disk does not prove the active PyTorch build supports it.

Record the GPU model, driver version, operating system, runtime package version, and the output of the framework's availability check. Avoid reinstalling every CUDA component at once; compatibility problems become harder to isolate when all layers change together.

02

Measure memory before lowering quality

Out-of-memory errors depend on model size, precision, batch size, resolution, attention implementation, and other processes using the device. Capture free memory immediately before the failing action and compare a known smaller workload. This distinguishes a capacity ceiling from fragmentation or a leak across repeated runs.

Reduce one demand variable at a time: batch size first, then resolution or sequence length, then optional models or upscalers. Close unrelated GPU processes. Restarting can clear retained allocations, but a repeatable workload that still exceeds capacity needs a durable configuration change.

  • Record baseline and peak VRAM usage.
  • Test batch size one with a known model and input.
  • Confirm whether memory grows across repeated runs.
  • Use mixed precision only when the model and hardware support it.
03

Diagnose container GPU access separately

A Docker error about selecting a device driver with GPU capabilities occurs before the application can allocate CUDA memory. Confirm the host driver, NVIDIA Container Toolkit, Docker runtime configuration, and Compose GPU reservation. Then test a minimal vendor image before testing the full application.

Do not install a second driver inside the application container. Containers normally use the host kernel driver while carrying compatible user-space libraries. Mixing host and container responsibilities is a common source of brittle images.

04

Keep a compatibility record

Once a working combination is found, record the driver, image digest or package versions, model, precision, and launch flags. Pin application dependencies and preserve a small smoke test that performs one real GPU operation.

Upgrade one layer at a time and rerun the smoke test. This turns future CUDA errors into a comparison against a known baseline instead of another full-stack reinstall.
