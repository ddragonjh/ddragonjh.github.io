Container failures from Docker to Kubernetes

Trace image, startup, port, storage, memory, and health failures in the order they occur.

This guide favors observable evidence, reversible tests, and official product documentation. Commands are examples; adapt names and paths to the environment you are responsible for.

01

Follow the container lifecycle

A container must pull an image, create a filesystem and network, start a process, remain healthy, and expose the expected service. Diagnose the earliest failed stage. ImagePullBackOff occurs before application startup; CrashLoopBackOff means a process started and repeatedly exited; a failing readiness probe can hide a running application from traffic.

In Kubernetes, events explain scheduling and image problems while current and previous logs explain application exits. In Docker, inspect the daemon state, container status, port bindings, and logs in that order. Mixing evidence from different restarts can send the investigation toward an old failure.

docker ps -adocker logs CONTAINERkubectl describe pod PODkubectl logs POD --previous
02

Separate image access from image contents

An image pull failure is usually a name, tag, registry authentication, network, or architecture problem. It is not an application configuration problem because the application has not started. Copy the exact image reference from the workload and test access from the environment that actually pulls it.

Once the image is present, inspect its entrypoint, command, working directory, and required files. A locally working image can still fail in a cluster if it assumes a bind-mounted file, host architecture, or shell that is not present in production.

  • Confirm repository, tag, registry, and imagePullSecret.
  • Check node architecture and image manifest platforms.
  • Keep registry credentials out of public logs.
03

Read termination state before changing limits

Exit code 137 often accompanies SIGKILL and may indicate an out-of-memory kill, but the termination reason is stronger evidence than the number alone. Compare actual memory usage with requests and limits, then determine whether the process leaks, caches too much, or legitimately needs more memory.

Increasing a limit without measuring usage can move the failure to the node. Conversely, optimizing an application will not help when a limit is simply below its normal startup requirement. Capture peak usage and repeat with one controlled workload.

kubectl top pod PODkubectl get pod POD -o yamldocker stats --no-stream
04

Treat ports, disks, and health checks as host boundaries

Port conflicts occur on the host side of a mapping. Disk errors can involve bytes, inodes, writable layers, logs, or volumes. Health checks run from a defined network location and may fail even when a browser on the host succeeds.

Inspect before deleting. Record which process owns a port, which Docker object consumes storage, and which exact probe request failed. Cleanup should target confirmed unused resources and preserve named volumes unless their removal is intentional and backed up.
